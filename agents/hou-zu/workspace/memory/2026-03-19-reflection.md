# 反思记录：openclaw.json 旧版本字段遗留问题

**日期**: 2026-03-19  
**问题**: openclaw.json 存在旧版本遗留字段（diagnostics/browser），导致 Guan-guan Heartbeat 任务报错

---

## 一、问题根因分析

### 1.1 直接原因
- `diagnostics` 和 `browser` 字段是 OpenClaw 早期版本的配置项
- 当前版本 (2026.3.13) 的配置 schema 已不再支持这些字段
- 升级过程中未清理旧版本遗留字段，导致解析失败

### 1.2 深层原因
1. **升级流程不完整**: 手动修改配置时未参考新版 schema，只关注需要的新功能
2. **配置变更无版本记录**: 缺乏配置变更历史，不清楚哪些字段是何时添加/废弃的
3. **缺乏配置校验**: 修改配置后未进行有效性校验，未发现不兼容字段
4. **文档缺失**: 未记录每个配置项的版本要求和使用场景

---

## 二、预防措施

### 2.1 升级前检查
- [ ] 查阅新版配置 schema 或 changelog
- [ ] 使用 `openclaw doctor` 检查配置兼容性
- [ ] 备份原配置文件

### 2.2 配置版本管理
- [ ] 配置文件纳入版本控制 (Git)
- [ ] 记录每次配置变更的原因和版本
- [ ] 使用配置模板而非手动创建

### 2.3 自动化校验
- [ ] 部署前运行配置校验
- [ ] 心跳任务增加配置健康检查
- [ ] 关键配置变更前先在测试环境验证

---

## 三、经验沉淀

### 3.1 配置管理最佳实践
| 场景 | 推荐做法 |
|------|----------|
| 新功能配置 | 先查阅官方文档 |
| 升级主版本 | 先备份，再运行 doctor 检查 |
| 手动修改 | 记录变更日志，注明版本号 |
| 批量部署 | 使用配置模板 + 环境变量 |

### 3.2 关键教训
1. **不要手动移植旧配置** - 升级时应使用新版默认配置作为基础
2. **保留变更记录** - 每次修改配置都要有文档说明
3. **先验证再部署** - 重要任务前用 `openclaw doctor` 检查健康状态
4. **关注错误信息** - 类似 "unknown field" 的警告不能忽视

### 3.3 相关命令
```powershell
# 检查配置健康
openclaw doctor

# 查看配置版本
openclaw --version

# 备份配置
Copy-Item openclaw.json openclaw.json.bak
```
