# 定时任务创建规范

> 所有 Agent 创建定时任务必须遵守本规范

---

## 一、创建流程（跨 Agent 协作）

```
[任意Agent]提出需求 → @管管审批 → [申请人]创建任务 → [申请人]上线通知
```

### 1. 提出需求
任意 Agent（如小艺、面小助等）发送消息到群里 @管管，申请创建定时任务，必须提供：
- **任务名称**：清晰描述任务目的
- **执行频率**：cron 表达式或间隔
- **执行内容**：完整的 prompt
- **超时时间**：timeoutSeconds
- **预期产出**：任务执行后应该产出什么
- **监控方式**：announce / none

### 2. 审批
管管在群里收到申请后：
- 检查任务配置合理性
- 确认 delivery 模式设置正确
- 发送审批结果到群里 @申请人
- 审批通过后，告知申请人可以创建

### 3. 创建
**申请人**收到审批通过后，自行创建任务

### 4. 上线成功通知
**申请人**在任务创建成功后，在群里通知：
- 任务已上线
- 首次执行时间
- 任务ID

---

## 二、Delivery 模式选择

| 场景 | mode | 说明 |
|------|------|------|
| **重要业务任务** | `announce` | 每次执行后发送报告到聊天渠道 |
| **心跳/进化任务** | `none` | 仅内部执行，不打扰用户 |
| **Webhook 回调** | `webhook` | 外部系统集成 |

**注意**：
- 除心跳/进化类任务外，**其他任务一律使用 `announce`**
- `announce` 必须指定 `channel` 和 `to`

---

## 三、监控指标

创建任务后必须关注：

| 指标 | 阈值 | 处理 |
|------|------|------|
| `consecutiveErrors` | > 0 | 立即排查 |
| `lastRunStatus` | error | 记录错误，检查配置 |
| `lastDurationMs` | 接近 timeout | 考虑增加超时时间 |
| `nextRunAtMs` | 超过预期时间 | 检查调度器状态 |

---

## 四、错误处理

### 连续错误告警
- `consecutiveErrors >= 3`：自动禁用任务并通知
- `consecutiveErrors = 1-2`：排查原因，修复后重新测试

### 常见错误类型
1. **语法错误**：cron 表达式格式错误
2. **超时**：timeoutSeconds 设置过短
3. **权限**：缺少必要的 API 权限
4. **资源**：文件路径不存在、依赖缺失

## 五、文档记录

每个定时任务必须记录以下信息：

```markdown
## 任务ID: xxx

### 基础信息
- Agent: 
- 任务名称: 
- 创建时间: 
- 创建人: 

### 配置
- 调度: 
- 超时: 
- 投递模式: 

### 测试记录
- 测试时间: 
- 测试结果: 
- 调整: 

### 执行状态
- 最后运行: 
- 最后状态: 
- 连续错误: 
```

---

## 六、审批检查清单

创建前管管会检查：

- [ ] 是否已手动测试 1-3 次？
- [ ] 测试结果是否记录？
- [ ] cron 表达式是否正确？
- [ ] timeoutSeconds 是否合理？（建议 2-5 倍预估）
- [ ] delivery 模式是否匹配场景？
- [ ] 错误处理机制是否考虑？
- [ ] 是否需要通知？通知谁？

---

## 七、禁止事项

❌ **禁止**未经测试直接创建定时任务  
❌ **禁止**将 `announce` 设为 `none`（重要任务）  
❌ **禁止**不记录测试结果就上线  
❌ **禁止**创建后不监控直接离开  

---

*本规范由管管制定，所有 Agent 创建定时任务必须遵守*
