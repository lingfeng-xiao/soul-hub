# 定时任务创建工作流规范

> 所有 Agent 创建定时任务必须遵守本规范
>
> **核心原则**：未经充分测试的任务不允许上线
>
> **⚠️ 注意事项**：
> - subagent 没有飞书 API 权限，必须使用 message 工具
> - exec 创建 cron 可能超时，但任务会实际创建成功

---

## 一、工作流定义

```
[申请人]提出需求 → @管管审批 → [管管]创建任务 → [申请人]上线通知
```

| 步骤 | 执行者 | 工具 |
|------|--------|------|
| 1. 提出需求 | 申请人 | message |
| 2. 审批 | 管管 | message |
| 3. 创建任务 | 管管 | exec (openclaw cron create) |
| 4. 上线通知 | 申请人 | message |

**注意**：申请人（subagent）没有 exec 创建 cron 的权限，必须由管管代为创建。

---

## 二、测试验证规范

### 测试阶段

```
需求确认 → 单元测试 → 集成测试 → 审批 → 上线
```

#### 1. 需求确认（申请人）
- 明确任务目标
- 确定执行频率
- 编写 prompt
- 预估执行时间和资源需求

#### 2. 单元测试（申请人）
- 测试核心功能是否正常
- 验证 API 调用是否成功
- 检查参数是否正确
- **至少测试 1-3 次**

#### 3. 集成测试（申请人）
- 测试完整执行流程
- 验证输出是否符合预期
- 检查错误处理是否健壮

#### 4. 审批（管管）
- 检查配置是否合理
- 确认测试记录完整
- 返回审批结果

#### 5. 上线（申请人）
- 创建任务（exec 可能超时但会成功）
- 验证任务已创建
- 发送上线通知

---

## 三、审批检查清单

管管审批时检查：

- [ ] 测试是否完成 1-3 次？
- [ ] 测试结果是否记录？
- [ ] cron 表达式是否正确？
- [ ] timeout 是否合理？
- [ ] delivery 模式是否匹配场景？
- [ ] 错误处理是否考虑？
- [ ] 资源占用是否合理？

## 五、工具使用规范

### 发消息：必须使用 message 工具

```
message action=send channel=feishu message="消息内容" target="群ID"
```

**禁止使用**：
- feishu_im_user_message（没有权限）
- feishu_get_user（没有权限）

### 创建定时任务：使用 exec 工具

```
openclaw cron create --name "任务名" --cron "*/30 * * * *" --message "执行内容" --timeout-seconds 10 --announce --channel feishu --to "群ID"
```

**注意**：exec 可能超时，但任务会实际创建成功。超时后检查 cron list 确认任务是否已创建。

### 会话管理

工作流执行完成后，需要清理产生的临时会话：

- 使用 `sessions_list` 查看当前会话
- 使用 `subagents` 或手动清理不需要的会话
- 避免会话泄漏导致资源浪费

## 六、禁止事项

❌ **禁止**未经测试直接申请审批  
❌ **禁止**使用飞书 API 发消息（用 message）  
❌ **禁止**审批未通过就创建任务  
❌ **禁止**创建后不上线通知  
❌ **禁止**不记录审计日志就创建任务

## 七、特殊情况

### 审批拒绝后
- 申请人修改配置
- 重新进行单元测试
- 重新提交审批

### 创建超时
- 检查 cron list 确认任务是否已创建
- 如果已创建，发送上线通知
- 如果未创建，重新执行 exec

### 任务异常
- 申请人负责排查和修复
- 连续失败 3 次以上可申请禁用
