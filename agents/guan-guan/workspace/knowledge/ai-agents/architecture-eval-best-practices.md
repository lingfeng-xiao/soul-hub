## 核心要点一：构建有效 Agent 的原则

### 1.1 架构选择：Workflows vs Agents

| 类型 | 特点 | 适用场景 |
|------|------|----------|
| **Workflows** | 预定义代码路径 orchestration | 任务明确、需可预测性 |
| **Agents** | LLM 动态指导流程，保持控制 | 灵活性、规模化、模型驱动决策 |

### 1.2 关键洞察

> 最成功的实现使用**简单、可组合的模式**，而不是复杂框架。

- 复杂框架增加抽象层，掩盖底层 prompts 和 responses
- 建议：从 LLM API 直接开始，许多模式几行代码即可实现
- 如需框架，确保理解底层代码

### 1.3 基础构建块：增强型 LLM

Augmented LLM = LLM + 检索 + 工具 + 记忆

实现建议：
- 针对具体用例定制这些能力
- 提供简单、文档良好的接口
- 推荐 Model Context Protocol (MCP)

### 1.4 核心工作流模式

#### Prompt Chaining
- 将任务分解为顺序步骤
- 每个 LLM 调用处理前一个输出
- 可添加程序化检查（gate）
- **适用**：任务可清晰分解为固定子任务

#### Routing（路由）
- 分类输入并导向专门的 followup 任务
- 允许分离关注点，构建更专业的 prompts
- **适用**：复杂任务有不同类别，分开处理更好

#### Parallelization（并行化）
- Sectioning：分解为独立子任务并行运行
- Voting：多次运行获取多样化输出
- **适用**：可分解的独立子任务或需多样化结果

## 核心要点二：AI Agent 评估框架

### 2.1 评估结构

```
Task (任务) → Trial (试验) → Grader (评分器) → Transcript (转录) → Outcome (结果)
```

| 概念 | 定义 |
|------|------|
| **Task** | 单个测试，有定义的输入和成功标准 |
| **Trial** | 每次尝试（因模型输出有变异，多次运行更稳定） |
| **Grader** | 评分逻辑，一个任务可有多个 graders，每个包含多个 assertions |
| **Transcript** | 完整记录（输出、工具调用、推理、中间结果） |
| **Outcome** | 试验结束时的最终环境状态 |

### 2.2 评估复杂度

**为什么 Agent 评估更困难？**
- 多轮交互：调用工具、修改状态、根据中间结果调整
- 错误传播：错误可以传播和累积
- 创造性解决方案：前沿模型可能找到超出静态评估预期的方法

### 2.3 评估的价值

- 在生产前发现问题
- 让问题和行为变化可见
- 随着 Agent 生命周期价值累积
- 帮助团队更有信心地发布

### 2.4 实践建议

- **早期**：Evals 迫使产品团队明确"成功"是什么
- **后期**：帮助维持一致的质量标准
- 结合：生产监控、A/B 测试、用户研究

## 应用到管管

### 当前状态
- 管管是 Agent 管理员，负责 Agent 注册管理
- 使用飞书工具集进行任务、日程、多维表格管理

### 可改进点
1. **工作流模式**：对于复杂任务（如批量创建 Agent），可采用 Prompt Chaining 分解步骤
2. **评估机制**：目前缺少系统化的自检机制，可建立更规范的检查流程
3. **框架选择**：保持简单，不盲目引入复杂框架
