# AI Agent 基础设施噪声与评估

> 来源: Anthropic Engineering Blog (2026-03)
> 主题: 基础设施配置对 Agentic Coding Evals 的影响

## 核心发现

**关键数字:**
- 资源配置差异可以导致 **6 个百分点** 的分数变化 (p < 0.01)
- 在某些情况下，这个差异超过排行榜顶级模型的差距

## 问题背景

传统静态 benchmark 直接评分模型输出，运行时环境不影响结果。

Agentic coding evals 不同：
- 模型获得完整环境
- 需要写程序、运行测试、安装依赖、多轮迭代
- 运行时成为问题解决过程的组成部分

## 资源限制的影响机制

### 两种资源配置参数
1. **Guaranteed allocation**: 预先保留的资源
2. **Hard limit (kill threshold)**: 容器被杀的阈值

### 关键问题
当两者设置为相同值时，没有瞬时峰值空间：
- 短暂的内存波动会导致 OOM kill
- 容器被杀死，但不是因为模型无法解决问题

## 实验结果

### Terminal-Bench 2.0 实验
| 配置 | 基础设施错误率 | 成功率变化 |
|------|--------------|----------|
| 1x (严格) | 5.8% | 基准 |
| 3x headroom | 2.1% | 在噪声范围内 |
| Uncapped | 0.5% | +6% |

### 关键发现
- **3x 以下**: 额外资源主要修复基础设施可靠性问题
- **3x 以上**: 额外资源开始帮助 agent 解决原本解决不了的问题
- 这意味着资源限制改变了评估的内容

## 对评估的影响

### 不同资源策略奖励不同
- **紧张限制**: 奖励高效、精简的策略
- **宽松限制**: 奖励能更好利用资源的 agent

### 示例: bn-fit-modify 任务
- 某些模型第一步是安装标准 Python 数据科学栈
- 宽松限制: 可以成功
- 紧张限制: 安装时就 OOM，连代码都没写

## 其他方差来源

- **时间限制**: 在某些配置下开始发挥作用
- **集群健康**: 硬件规格、并发级别、甚至出口带宽
- **API 延迟**: 随时间波动（白天/晚上）

## 建议

### 对评估维护者
1. 指定 **两个参数**: guaranteed allocation 和 hard limit（不是单一固定值）
2. 校准两者之间的带宽，使 floor 和 ceiling 的分数在噪声范围内
3. 对于 Terminal-Bench 2.0，3x ceiling 是合理的平衡点

### 对模型消费者
1. 不要盲目相信排行榜的微小差距
2. 资源配置是重要的混淆因素

## 核心教训

> Agentic evals 是端到端的系统测试。任何组件都可能成为混淆因素。

"模型能力"和"基础设施行为"之间的边界比单一 benchmark 分数显示的更模糊。
