# Agent 架构选择决策模板

> 基于 Anthropic 最佳实践
> 日期: 2026-03-18

## 决策流程

```
1. 任务复杂度评估
       ↓
2. 选择架构模式
       ↓
3. 确定 Agent 数量
       ↓
4. 设计工具接口
```

---

## 1. 任务复杂度评估

| 复杂度 | 特征 | 推荐方案 |
|--------|------|----------|
| **简单** | 单次查询，直接回答 | 增强型 LLM（无需 Agent） |
| **中等** | 多步骤，有明确流程 | Workflow（Prompt Chaining/Routing） |
| **复杂** | 开放性，需要灵活决策 | Agent 或 Orchestrator-Workers |
| **超复杂** | 多方面研究，超大信息量 | Multi-Agent 系统 |

---

## 2. 架构模式选择

### 单 LLM 增强 (基础)

**何时使用**：
- 任务可以直接完成
- 需要检索增强
- 只需要简单工具

**实现**：
```
LLM + 检索 + 工具 + 记忆
```

### Workflow 模式

#### Prompt Chaining
**何时**：任务可分解为固定子任务
**示例**：写大纲 → 检查 → 写正文

#### Routing
**何时**：有明确的分类，需要分别处理
**示例**：客服分流（退款/技术/咨询）

#### Parallelization
**何时**：子任务相互独立，或需要多视角
**示例**：代码审查（多人审查）

#### Evaluator-Optimizer
**何时**：有明确评估标准，需要迭代
**示例**：翻译优化、搜索优化

### Agent 模式

**何时**：
- 任务步骤无法提前预测
- 需要模型动态决策
- 需要长时间自主操作

### Multi-Agent 模式 (高级)

**何时**：
- 需要并行探索多个方向
- 信息量超过单 Agent 上下文
- 需要专业分工

**性能收益**：+90.2%（Anthropic 数据）
**成本代价**：~15x Token 消耗

---

## 3. Agent 数量规划

| 任务类型 | Agent 数量 | 工具调用上限 |
|----------|------------|--------------|
| 简单事实查找 | 1 | 3-10 次 |
| 直接比较 | 2-4 | 10-15 次 |
| 复杂研究 | 10+ | 视情况 |

---

## 4. 工具接口设计

### 原则

1. **清晰描述** - 每个工具功能明确
2. **匹配意图** - 工具与任务目标一致
3. **适度数量** - 避免过多选择导致决策疲劳

### 检查清单

- [ ] 每个工具有明确的输入/输出格式
- [ ] 工具描述包含使用场景
- [ ] Agent 能访问所有必要工具
- [ ] 工具之间无功能重叠

---

## 5. 成本-收益分析

```
价值评估 = 性能提升 / (延迟增加 × 成本增加)
```

### 阈值建议

- **价值 > 10x 成本**：使用 Multi-Agent
- **价值 > 3x 成本**：使用单 Agent
- **价值 < 3x 成本**：使用 Workflow 或单 LLM

---

## 飞书场景应用

| 场景 | 推荐模式 | 理由 |
|------|----------|------|
| 日程查询 | 单 LLM | 简单直接 |
| 会议安排 | Routing + Agent | 需要判断+多步操作 |
| 多表格汇总 | Orchestrator-Workers | 多数据源 |
| 复杂研究 | Multi-Agent | 并行探索 |

---

*使用说明：根据任务特征从左到右决策，最终确定架构模式*
