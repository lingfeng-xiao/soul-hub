# AI Agent 评估完全指南

> 来源：Anthropic Engineering Blog - 2026年1月
> 作者：Mikaela Grace, Jeremy Hadfield, Rodrigo Olivares, Jiri De Jonghe

## 评估的基本结构

- **Task（任务）**：单个测试，有定义的输入和成功标准
- **Trial（试验）**：每次尝试，因为模型输出在运行之间会有变化
- **Grader（评分器）**：评分逻辑，一个任务可以有多个 graders
- **Transcript（记录）**：完整记录，包括输出、工具调用、推理等
- **Outcome（结果）**：环境在试验结束时的最终状态

## 三种评分器类型

| 类型 | 优点 | 缺点 |
|------|------|------|
| 代码评分 | 快速、便宜、客观、可复现 | 脆弱、缺乏细微差别 |
| 模型评分 | 灵活、可扩展、处理开放任务 | 非确定性、更昂贵 |
| 人工评分 | 金标准质量 | 昂贵、缓慢 |

## 两种评估目标

1. **Capability evals（能力评估）**：问"这个 Agent 能做什么？"
   - 初始通过率应较低，针对 Agent 擅长的任务
   - 给团队一座需要攀登的山

2. **Regression evals（回归评估）**：问"Agent 是否仍然处理所有曾经能处理的任务？"
   - 应该有接近 100% 的通过率
   - 防止倒退，得分下降意味着有问题需要修复

## 评估器类型与评估方法

### 编码 Agent
- 使用确定性测试验证结果
- SWE-bench Verified：运行测试套件
- Terminal-Bench：端到端技术任务

### 对话 Agent
- 需要 LLM 模拟用户
- 多维度成功：状态检查 + transcript 约束 + LLM 评分

### 研究 Agent
- groundedness checks：验证声明是否有来源支持
- coverage checks：定义好答案必须包含的关键事实
- LLM 评分需与人类专家校准

## 评估的非确定性

- **pass@k**：k 次尝试中至少一次成功的概率
- **pass^k**：所有 k 次试验都成功的概率
- 选择取决于产品需求：pass@k 用于工具类（一次成功即可），pass^k 用于需要一致性的场景

## 实用建议

1. **从小做起**：20-50 个简单任务即可开始
2. **构建无歧义任务**：好的任务应该是两个领域专家能独立得出相同的通过/失败结论
3. **检查 transcript**：定期阅读失败案例，判断是 Agent 真正犯错还是评分器问题
4. **监控评估饱和**：100% 通过率追踪回归但不提供改进信号
5. **避免一次性评估**：运行多个 trials 以获得更稳定的结果
