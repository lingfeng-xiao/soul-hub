# Agentic Evals 基础设施噪声：资源配额如何影响评估结果

> 来源: Anthropic Engineering Blog (2026-03)
> 主题: 量化基础设施配置对 AI Agent 编码评估的影响

---

## 核心发现

**资源配置可以导致评估分数波动达 6 个百分点** — 有时甚至超过排行榜榜首之间的差距。

这是一个关键发现：静态基准直接评分模型输出，但 **Agentic Coding Evals** 不同 — 模型在完整环境中写程序、运行测试、安装依赖、多轮迭代。运行时不再是被动容器，而是问题解决过程的组成部分。

---

## 实验数据

在 Terminal-Bench 2.0 上的实验结果：

| 配置 | 基础设施错误率 | 成功率提升 |
|------|--------------|-----------|
| 1x (严格限制) | 5.8% | 基准 |
| 3x (3倍余量) | 2.1% | 在噪声范围内 |
| 无限配额 | 0.5% | +6 个百分点 |

- 从 1x 到 3x：基础设施错误率大幅下降（5.8% → 2.1%，p < 0.001），但成功率在噪声范围内
- 从 3x 到无限：成功率上升约 4 个百分点，超过了基础设施改善的幅度
- 无限资源配置下，总提升 +6 个百分点（p < 0.01）

---

## 关键机制

### 为什么资源会影响评估？

1. **低于 3x**：额外资源主要修复基础设施可靠性问题（瞬态资源峰值）
   - 沙箱提供者隐式允许临时过量分配
   - 评估变得更稳定，但没有变容易

2. **超过 3x**：额外资源开始**主动帮助 Agent 解决问题**
   - 资源限制改变了评估的内容
   - 宽松限制奖励"暴力"方案，紧凑限制奖励"高效"方案

### 具体例子

在 bn-fit-modify 任务中（贝叶斯网络拟合）：
- **宽松限制**：可以安装完整的 Python 数据科学生态（pandas, networkx, scikit-learn）
- **紧凑限制**：Pod 在安装期间就内存不足，Agent 连一行代码都没写就失败了

---

## 对测量意味着什么

| 资源限制 | 评估的是什么 |
|---------|-------------|
| 紧凑 (1x) | Agent 的高效策略能力 |
| 宽松 (无限) | Agent 的全面解决问题能力 |

**两种都是合法的测试**，但如果不指定资源配置就collapse为单一分数，很难解释差异和现实世界的泛化能力。

---

## 其他方差来源

资源配额不是唯一的隐藏变量：

- **时间限制** — 在某些配置下开始发挥作用
- **集群健康** — 硬件规格、并发级别
- **出口带宽** — API 延迟随流量模式变化
- **时间因素** — 通过观察发现通过率随一天中的时间波动

> "模型能力"和"基础设施行为"之间的边界比单一基准分数暗示的更加模糊。

## 推荐做法

### 1. 双重资源配置规范

```yaml
# 不要这样做（单一值 = Guaranteed = Kill Threshold）
resources:
  memory: "4Gi"  # 零余量，瞬态峰值会导致 OOM

# 应该这样做（分离两个参数）
resources:
  guaranteed: "4Gi"    # 预留资源
  limit: "12Gi"        # 3x 余量，允许瞬态峰值
```

### 2. 校准原则

- 分数在 floor 和 ceiling 之间应该在噪声范围内
- 对于 Terminal-Bench 2.0，3x ceiling 是合理平衡点
- 具体倍数因基准和任务分布而异，应报告

### 3. 对于基准维护者

- 发布推荐资源配置（已有 Terminal-Bench 2.0）
- **发布执行方法论** — 填补发现的差距
- 在多个时间和多个日期运行有助于平均化噪声

---

## 实践意义

对于使用基准分数做决策的人：

> "今天，排行榜上 2 点的领先可能反映真正的能力差异，或者只是因为一个评估运行在更强的硬件上，或者在一天中更幸运的时间，或两者兼有。"

**关键结论**：
1. Agentic Evals 的资源配置应被视为**一等实验变量**
2. 文档和控制应与 prompt 格式或采样温度一样严格
3. 排行榜差距可能包含大量基础设施噪声
