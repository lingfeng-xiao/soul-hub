# 多 Agent 研究系统的架构与性能洞察

> 来源: Anthropic Engineering Blog (2025-06-13)
> 主题: 多 Agent 协作系统

## 核心发现

多 Agent 研究系统在广度优先查询中表现优异，比单 Agent 性能提升 **90.2%**。

## 架构模式：Orchestrator-Worker

```
User Query → Lead Agent → Spawns Subagents (parallel) → Results → Lead Agent → Final Answer
```

- **Lead Agent**：分析查询、制定策略、协调子 Agent
- **Subagents**：并行探索不同方面，使用独立上下文窗口

## 性能关键因素

| 因素 | 解释方差 |

| Token 使用量 | **80%** |

## Token 消耗对比

| 系统类型 | Token 倍数 |
|----------|------------|
| 单 Agent 聊天 | 1× |
| 单 Agent 研究 | ~4× |
| 多 Agent 系统 | ~15× |

## 适用场景 ✅

- 高价值任务（价值 > 增加的成本）
- 大量并行化
- 信息超过单个上下文窗口
- 接口众多复杂工具

## 不适用场景 ❌

- 需要所有 Agent 共享同一上下文
- 任务间有强依赖关系
- 实时协调要求高（如多数编码任务）

## 设计原则

1. **并行化优先**：让子 Agent 同时探索不同方向
2. **分离关注点**：每个子 Agent 独立工具、提示、探索轨迹
3. **上下文管理**：长上下文时保存计划到 Memory，防止截断
4. **经济性权衡**：多 Agent 成本高，确保任务价值足够
