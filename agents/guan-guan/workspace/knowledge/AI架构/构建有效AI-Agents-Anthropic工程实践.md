## 核心观点

**最简单的解决方案往往是最有效的。** 成功的 AI Agent 实现使用的是简单、可组合的模式，而不是复杂的框架。

---

## Workflows vs Agents

### Workflows（工作流）
- 通过**预定义代码路径**编排 LLM 和工具
- 适用于**明确、确定性**的任务
- 优点：可预测、一致性强

### Agents（代理）
- LLM **动态指导**自己的流程和工具使用
- 适用于**开放、需要灵活性**的任务
- 优点：自主决策、适应性强的

---

## 五大工作流模式

### 1. Prompt Chaining（提示链）
- 将任务分解为**固定顺序**的步骤
- 每个 LLM 处理前一个的输出
- 适用场景：营销文案生成→翻译、写作大纲→写文档

### 2. Routing（路由）
- 对输入进行**分类**，导向专门的下游任务
- 分离关注点，建立更专业的 prompts
- 适用场景：客服分类（问题/退款/技术支持）、简单问题→小模型，复杂问题→大模型

### 3. Parallelization（并行化）
- 独立子任务**并行**运行，或多次运行**投票**
- Sectioning：分割独立任务
- Voting：多次运行获取多样性
- 适用场景：Guardrails、内容审核、代码漏洞审查

### 4. Orchestrator-workers（编排器-工作者）
- 中央 LLM **动态分解**任务，委托给工作 LLM，合成结果
- 关键：子任务**不预定义**，由编排器根据输入决定
- 适用场景：复杂编码任务、多源搜索分析

### 5. Evaluator-optimizer（评估器-优化器）
- 一个 LLM 生成响应，另一个提供评估和反馈，**循环迭代**
- 适用场景：文学翻译、复杂搜索任务

---

## Agents 最佳实践

### 何时使用 Agents
- 任务**无法预定义**步骤数量
- 需要**模型驱动**的决策能力
- 需要在**可信环境**中扩展任务

### 核心原则
1. **保持简单** - 设计简单优先
2. **透明性** - 展示 agent 的规划步骤
3. **ACI (Agent-Computer Interface)** - 清晰工具文档和测试

### 实现要点
- 从人类命令或交互式讨论开始
- 执行过程中从环境获取"真实结果"评估进度
- 在检查点或障碍时暂停等待人类反馈
- 设置停止条件（如最大迭代次数）

## 框架使用建议

### 推荐框架
- Claude Agent SDK
- AWS Strands Agents
- Rivet (GUI)
- Vellum (GUI)

### 注意事项
- 框架简化了底层任务（调用 LLM、定义工具、链式调用）
- 但增加了**抽象层**，可能掩盖 prompts 和响应
- 容易引入不必要的复杂度
- **建议**：从简单 API 开始，几行代码就能实现很多模式

---

## 三大核心原则（总结）

```
1. Simplicity - 保持简单
2. Transparency - 透明性（展示规划步骤）
3. ACI - Agent-Computer Interface（工具文档和测试）
```
