# 构建高效 AI Agents 的核心模式

> 来源: Anthropic Engineering Blog - Building Effective Agents (2024.12)
> 核心观点: 简单可组合的模式 > 复杂框架

---

## 一、核心认知

### 1.1 Workflow vs Agent 区分

| 类型 | 定义 | 适用场景 |
|------|------|----------|
| **Workflow** | 预定义代码路径编排的 LLM + 工具 | 任务明确、流程固定 |
| **Agent** | LLM 动态指挥自己的流程和工具使用 | 需要灵活性、模型自主决策 |

### 1.2 关键原则

> **最简单的解决方案优先**
> - 单次 LLM 调用 + retrieval + in-context examples 通常足够
> - Agent 往往用延迟和成本换取更好的任务表现
> - 不要为了" agentic"而使用 agent

---

## 二、构建块: Augmented LLM

**增强型 LLM = LLM + Retrieval + Tools + Memory**

核心实现要点：
1. 针对具体用例定制这些能力
2. 确保提供简单、文档良好的接口

> MCP (Model Context Protocol) 是集成第三方工具的简洁方案

---

## 三、五大工作流模式

### 3.1 Prompt Chaining (链式)

**结构**: 任务 → 步骤1 LLM → 步骤2 LLM → ... → 输出

**何时使用**: 任务可干净地分解为固定子任务，用延迟换取更高准确率

**示例**:
- 写营销文案 → 翻译成其他语言
- 写大纲 → 检查大纲 → 写文档

### 3.2 Routing (路由)

**结构**: 输入 → 分类器 → 专用下游任务

**何时使用**: 复杂任务有明确分类，分类可准确处理

**示例**:
- 客服问题分流 (一般/退款/技术支持)
- 简单问题 → 小模型，复杂问题 → 大模型

### 3.3 Parallelization (并行)

**结构**: 输入 → [并行任务1, 任务2, ...] → 聚合

**何时使用**: 子任务独立，可同时处理

**变体**:
- **Sectioning**: 分块处理 (如同时分析多份文档)
- **Voting**: 多版本投票 (如代码审查)

### 3.4 Orchestrator-Workers (编排器-工作器)

**结构**: 动态决定调用哪些工作器

**何时使用**: 任务步骤无法预定义，需要动态决策

### 3.5 Evaluator-Optimizer (评估器-优化器)

**结构**: 生成 → 评估 → 不满意则优化 → 循环

**何时使用**: 有明确评估标准，可以迭代改进

**示例**:
- 搜索任务优化
- 代码迭代改进

## 四、框架使用建议

### 推荐做法
- **从 LLM API 直接开始**: 很多模式几行代码就能实现
- 使用框架时，确保理解底层代码

### 常见陷阱
- 框架增加抽象层，隐藏 prompts 和 responses
- 容易过度设计，本可以用更简单方案

---

## 五、实践建议

### 起步
1. 从单次 LLM 调用开始
2. 只在需要时增加复杂度
3. 优先用 workflow，agent 留到需要灵活性的场景

### 调试
- 理解底层代码，不要假设框架行为
- prompts 和 responses 的可见性很重要
