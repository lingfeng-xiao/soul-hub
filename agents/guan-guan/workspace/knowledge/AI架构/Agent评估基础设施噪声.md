# Agent 评估中的基础设施噪声问题

> 来源: Anthropic Engineering Blog (2026-03)
> 原文: https://www.anthropic.com/engineering/infrastructure-noise

## 核心发现

**基础设施配置可以直接改变 Agent 编码评估的结果达 6 个百分点**——有时甚至超过排行榜上顶级模型之间的差距。

## 问题背景

- 传统基准测试：直接评分模型的输出，运行环境不影响结果
- Agentic 编码评估：模型在完整环境中编写程序、运行测试、安装依赖、多次迭代
- **关键差异**：运行时不再是被动容器，而是问题解决过程的组成部分

## 实验数据

在 Terminal-Bench 2.0 上的实验结果：

| 配置 | 基础设施错误率 | 成功率变化 |
|------|--------------|-----------|
| 严格资源限制 (1x) | 5.8% | 基准线 |
| 3x 资源头room | 2.1% | 仍在噪声范围内 |
| 完全不限制 | 0.5% | +6个百分点 (p<0.01) |

## 关键洞察

### 1. 资源限制改变评估内容

- **3x 以下**：额外资源修复的是基础设施可靠性问题（瞬时资源峰值）
- **3x 以上**：额外资源开始实际帮助 Agent 解决之前无法解决的问题
- **结论**：限制条件会改变评估衡量的内容

### 2. 资源策略与模型偏好

- **严格限制**：奖励高效、精简的代码策略
- **宽松限制**：奖励能充分利用资源的大型工具方案
- 两种策略都合法，但需要明确资源配置才能解释差异

### 3. 具体案例

在 Terminal-Bench 的 bn-fit-modify 任务中：
- 某些模型第一步是安装标准 Python 数据科学栈（pandas, networkx, scikit-learn）
- 宽松限制下：安装成功，开始解题
- 严格限制下：安装时就 OOM，根本没机会写代码

## 其他方差来源

资源分配不是唯一隐藏变量：
- 时间限制
- 集群健康状态
- 硬件规格
- 并发级别
- API 延迟（随流量模式和事件变化）

> "模型能力"和"基础设施行为"之间的边界比单一基准分数显示的更加模糊。

## 建议

### 对于评估开发者

1. **同时指定两个参数**：保证分配 + 硬性终止阈值，而不是单一固定值
2. **多次运行**：在不同时间和日期运行有助于消除噪声
3. **明确资源配置**：报告结果时必须说明资源设置

### 对于 Agent 开发者

1. **在受限环境测试**：确保 Agent 在资源受限情况下仍能工作
2. **考虑生产环境**：生产环境的资源设置应与评估一致
3. **多维度评估**：不要只看成功率，还要看资源效率

## 公式

```
实际测量 = 模型能力 + 基础设施行为 + 两者交互
```
