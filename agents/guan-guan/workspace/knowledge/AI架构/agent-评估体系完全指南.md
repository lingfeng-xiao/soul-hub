# Agent 评估体系完全指南

> 来源: Anthropic Engineering - Demystifying evals for AI agents (2026-01-09)

## 核心概念

### 评估基础元素

| 概念 | 定义 |
|------|------|
| **Task (任务)** | 单个测试，包含定义的输入和成功标准 |
| **Trial (测试)** | 每次尝试。由于模型输出有方差，通常运行多次以获得更稳定的结果 |
| **Grader (评分器)** | 评分逻辑，衡量 Agent 表现的某个方面。一个任务可有多个评分器 |
| **Transcript (轨迹)** | 完整的测试记录，包含输出、工具调用、推理过程、中间结果 |
| **Outcome (结果)** | 测试结束时的最终环境状态 |
| **Harness (测试框架)** | 端到端运行评估的基础设施 |

### Agent 特有组件

- **Agent Harness/Scaffold**: 使模型能够作为 Agent 行动的系统
  - 处理输入、编排工具调用、返回结果
  - 例如：Claude Code 是灵活的 Agent harness

- **Evaluation Suite**: 任务集合，测量特定能力或行为
  - 任务通常共享一个宽泛目标
  - 例如：客服评估套件可能包含退款、取消、转接等任务

## 为什么需要评估？

### 没有评估的问题

- 只能在生产环境发现问题（ reactive loop）
- 调试是被动的：等待投诉 → 手动复现 → 修复 → 希望没有其他回归
- 无法区分真实回归和噪音
- 无法在发布前自动测试数百个场景

### 评估的价值

- **问题可见化**：在影响用户前让问题和行为变化可见
- **价值累积**：随 Agent 生命周期价值递增
- **指导改进**：帮助识别问题、指导改进、聚焦研究和产品协作

## 评估复杂度层次

### 1. 单轮评估 (Single-turn)
- 简单：prompt → response → grading
- 适用于非 Agentic 的早期 LLM

### 2. 多轮评估 (Multi-turn)
- Agent 接收工具、任务、环境
- 执行 "agent loop"（工具调用 + 推理）
- 更新环境
- 使用单元测试验证

### 3. Agent 评估 (Agent Evaluation)
- 最复杂：跨多轮使用工具，修改环境状态
- 错误会传播和累积
- 前沿模型可能找到超出静态评估限制的创意方案

## 实践建议

### 构建评估的时机

- **早期**：强制产品团队明确"成功"意味着什么
- **后期**：帮助维持一致的质量标准

### 评估维度示例 (Descript)

三个成功维度：
1. Don't break things (不破坏功能)
2. Do what I asked (完成所求)
3. Do it well (做好)

### 评估演进路径

1. 人工评分 → LLM 评分器（由产品团队定义标准）→ 定期人工校准
2. 分离质量基准测试和回归测试套件

## 关键教训

> 评估的复杂性必须匹配被测系统的复杂性

- Agent 的能力（自治、智能、灵活）使其有用，但也使其更难评估
- 好的评估帮助团队更有信心地发布 AI Agent
