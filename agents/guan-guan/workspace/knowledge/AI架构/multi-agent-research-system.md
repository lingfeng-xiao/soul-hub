# Anthropic 多 Agent 研究系统实战

> 来源: https://www.anthropic.com/engineering/multi-agent-research-system
> 日期: 2026-03-18

## 核心成果

- 多 Agent 系统比单 Agent 性能提升 **90.2%**
- Token 使用量：约 **15 倍**于普通聊天
- 最佳场景：需要大量并行化的任务

## 为什么需要多 Agent 系统

### 研究任务的本质

- 开放性问题，难以提前预测所需步骤
- 过程动态且路径依赖
- 需要灵活性来 pivot 或探索切向连接
- 需要长时间自主操作

### 多 Agent 的优势

1. **并行探索不同方向**
2. **更好的信息压缩** - subagent 用独立上下文窗口探索不同方面
3. **分离关注点** - 减少路径依赖
4. **扩展性能** - 一旦智能达到阈值，多 Agent 是扩展性能的关键方式

## 架构模式：Orchestrator-Workers

```
用户查询 → Lead Agent → 并行创建 Subagents → 搜索/分析 → 返回结果 → Lead Agent 合成
```

### 工作流程

1. **Lead Agent** 分析查询，制定策略
2. **创建 Subagents** 执行特定研究任务
3. **Subagents** 并行使用搜索工具，迭代收集信息
4. **返回结果** 给 Lead Agent
5. **Lead Agent** 决定是否需要更多研究
6. **CitationAgent** 生成引用
7. **返回最终结果**

### 与传统 RAG 的区别

| 传统 RAG | 多 Agent 系统 |
|----------|--------------|
| 静态检索 | 动态多步搜索 |
| 一次性获取 | 适应新发现 |
| 简单匹配 | 分析性回答 |

## Prompt 工程最佳实践

### 1. 像 Agent 一样思考

- 迭代提示时，必须理解其效果
- 使用模拟测试，观察 Agent 逐步工作
- 立即发现失败模式

### 2. 教会编排器如何委托

每个 subagent 需要：
- **Objective** - 目标
- **Output format** - 输出格式
- **Tools and sources guidance** - 工具和来源指导
- **Clear task boundaries** - 清晰的任务边界

**教训**：简单指令如"研究芯片短缺"太模糊，导致 subagents 重复工作或误解任务。

### 3. 根据查询复杂度扩展工作量

嵌入明确的扩展规则：
- **简单事实查找**：1 个 agent，3-10 次工具调用
- **直接比较**：2-4 个 subagents，10-15 次调用
- **复杂研究**：10+ subagents，明确分工

### 4. 工具设计至关重要

- Agent-工具接口和 HCI 一样重要
- 使用正确的工具是高效的，常常是必需的
- 给 Agent 明确的启发式规则：
  - 先检查所有可用工具
  - 匹配工具使用与用户意图
  - 搜索网页进行广泛外部探索

## 性能分析

### Token 是关键

在 BrowseComp 评估中：
- **Token 使用本身解释了 80% 的性能差异**
- 工具调用次数
- 模型选择

### 成本与收益

- **成本**：多 Agent 约 15 倍于普通聊天的 Token
- **收益**：90.2% 性能提升
- **经济可行性**：任务价值必须足够高以支付增加的成本

## 适用场景

### ✅ 最佳场景

- 大量并行化的任务
- 超过单个上下文窗口的信息
- 与众多复杂工具交互

### ❌ 不适合场景

- 需要所有 Agent 共享同一上下文
- Agent 之间有大量依赖

## 飞书 Agent 落地建议

根据 Anthropic 经验，我们可以：

1. **学习路由模式** - 将任务分配给专门 Agent
2. **实现并行化** - 多个 Agent 同时处理不同方面
3. **建立评估循环** - Agent 产出 -> 评估 -> 优化
4. **平衡成本与价值** - 选择高价值任务使用多 Agent
