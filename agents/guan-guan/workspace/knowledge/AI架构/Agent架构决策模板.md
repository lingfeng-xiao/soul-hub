# Agent 架构决策模板

> 用于快速评估和选择合适的 Agent 架构模式

## 第一步：需求分析

### 任务特征
- [ ] 任务步骤可预先确定 → Workflows
- [ ] 需要灵活性、模型驱动决策 → Agents
- [ ] 两者混合 → Hybrid

### 性能要求
| 要求 | 建议 |
|------|------|
| 低延迟 | 避免 Agent，使用单次 LLM 调用 |
| 高准确性 | 可考虑 Workflows + 验证步骤 |
| 处理复杂输入 | 考虑 Routing |

### 成本敏感度
- [ ] 高 → 优先单次 LLM 调用
- [ ] 中 → Workflows
- [ ] 低 → Agents / Multi-Agent

## 第二步：架构选择

### 模式对比

```
┌─────────────────────────────────────────────────────────┐
│                    简单 → 复杂                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 单次 LLM 调用                                        │
│     ├── 检索增强 (RAG)                                    │
│     └── 上下文示例                                        │
│                                                         │
│  2. Workflows (预定义路径)                               │
│     ├── Prompt Chaining (串行)                          │
│     ├── Routing (分类跳转)                               │
│     ├── Parallelization (并行)                          │
│     └── Map-Reduce (映射归约)                            │
│                                                         │
│  3. Agents (动态决策)                                   │
│     ├── 单 Agent 循环                                    │
│     └── Multi-Agent (主从协调)                           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

---

## 第三步：检查清单

### 架构验证
- [ ] 是否从最简单的方案开始？
- [ ] 增加复杂度的理由是什么？
- [ ] 延迟/成本增加是否值得？

### 实现验证
- [ ] 是否可以直接用 LLM API 实现（不用框架）？
- [ ] 如果用框架，是否理解底层代码？
- [ ] 是否有错误处理和回退机制？

### 成本验证
- [ ] Token 消耗预估
- [ ] 多 Agent 场景：任务价值 > 15× 成本？

## 使用示例

### 示例 1：客服机器人
```
需求：分类处理不同类型查询
选择：Routing → 不同子工作流
理由：分离关注点，优化各类型处理
```

### 示例 2：深度研究
```
需求：探索复杂主题，多方向并行
选择：Multi-Agent (Orchestrator-Worker)
理由：需要大量并行化、token 预算充足
```

### 示例 3：内容生成
```
需求：生成后翻译
选择：Prompt Chaining
理由：任务清晰可分解，串行足够
```
