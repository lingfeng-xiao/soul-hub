# AI Agent 评估设计完全指南

> 来源: Anthropic Engineering Blog (2026-01-09)
> 原文: https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents

## 为什么需要评估

### 没有评估的问题

- 只能在生产环境发现问题，修复一个bug可能制造新bug
- 团队"盲目飞行"，无法验证除非猜测和检查
- 无法区分真实回归和噪声
- 无法在发布前针对数百个场景自动测试

### 评估的价值

- 让问题和行为变化在影响用户前可见
- 随着 Agent 生命周期价值累积
- 迫使产品团队明确"成功"的定义

## 评估的核心概念

### 基础术语

| 术语 | 定义 |
|------|------|
| **Task（任务）** | 具有定义输入和成功标准的单个测试 |
| **Trial（试次）** | 每次尝试，因为模型输出在运行之间有变化，需要多次试次 |
| **Grader（评分器）** | 评分 Agent 某些方面表现的逻辑，一个任务可有多个评分器 |
| **Transcript（转录）** | 试次的完整记录，包括输出、工具调用、推理、中间结果 |
| **Outcome（结果）** | 试次结束时环境中的最终状态 |
| **Harness（评估框架）** | 端到端运行评估的基础设施 |

### 评估层次

```
Evaluation Suite (评估套件)
  └── Tasks (任务)
        └── Trials (试次)
              └── Graders (评分器)
                    └── Assertions (断言)
```

## Agent 评估 vs 传统评估

### 单轮评估
- 提示 → 响应 → 评分
- 适合非 Agent 的早期 LLM

### 多轮评估
- Agent 接收工具、任务、环境
- 执行"Agent 循环"（工具调用 + 推理）
- 更新环境状态
- 使用单元测试验证结果

### Agent 评估的特殊性

1. **多 turn 操作**：跨多 turn 使用工具，修改状态
2. **错误传播**：错误可以传播和累积
3. **创造性解决方案**：前沿模型可能找到超越静态评估限制的方案

> 例如：Opus 4.5 在 τ2-bench 上发现了政策中的漏洞，"失败"了评估但实际给出了更好的用户解决方案。

## 构建评估的最佳实践

### 1. 多维度定义成功

参考 Descript 的做法，围绕三个维度构建评估：
- **不破坏东西** (Don't break things)
- **做我要求的** (Do what I asked)
- **做好它** (Do it well)

### 2. 渐进式构建

- **早期**：快速迭代 + 人工反馈
- **后期**：添加评估，先从狭窄区域开始（简洁性、文件编辑）
- **成熟后**：复杂行为评估（过度工程化）

### 3. 评估类型

| 类型 | 用途 | 频率 |
|------|------|------|
| 质量基准测试 | 衡量整体能力 | 定期 |
| 回归测试 | 确保变更不破坏功能 | 每次变更 |

### 4. Grader 设计

- 使用 LLM 作为评分器
- 由产品团队定义标准
- 定期人工校准

## 评估的组件关系

```
┌─────────────────────────────────────────────┐
│           Evaluation Harness                 │
│  (提供指令和工具、并发运行任务、记录、评分)   │
└─────────────────────────────────────────────┘
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
┌───────────────┐      ┌───────────────┐
│ Agent Harness │      │  Task Suite   │
│ (模型能作为   │      │ (任务集合)    │
│  Agent 运作)  │      └───────────────┘
└───────────────┘              │
        │                      ▼
        │              ┌───────────────┐
        │              │    Tasks      │
        │              │  (单个测试)   │
        │              └───────────────┘
        │                      │
        │                      ▼
        │              ┌───────────────┐
        │              │    Trials     │
        │              │  (多次试次)   │
        │              └───────────────┘
        │                      │
        └──────────┬───────────┘
                   ▼
          ┌───────────────┐
          │   Graderts    │
          │  (评分逻辑)   │
          └───────────────┘
```

## 评估 vs 生产监控

评估不是生产监控的替代品，而是互补：

| 评估 | 生产监控 |
|------|---------|
| 可控环境 | 真实用户场景 |
| 可重现 | 动态变化 |
| 开发时运行 | 持续运行 |
| 发现已知问题 | 发现未知问题 |

## 实用建议

1. **从狭窄开始**：先评估最关键的能力
2. **自动化**：集成到 CI/CD 流程
3. **多轮运行**：至少 10 次试次减少方差
4. **记录转录**：保存完整交互用于调试
5. **分层评估**：基础功能 + 复杂行为
