## 1. 自动循环引擎

### 循环控制器

```
当当前任务/循环完成时，自动执行:

LOOP_START:
  │
  ├─▶ [感知阶段] ──▶ 检查项目状态(git status, mvn compile)
  │                     │
  │                     ▼
  ├─▶ [分析阶段] ──▶ 有什么问题? 有什么机会?
  │                     │
  │                     ▼
  ├─▶ [规划阶段] ──▶ 决定下一步做什么
  │                     │
  │                     ▼
  ├─▶ [执行阶段] ──▶ 执行任务
  │                     │
  │                     ▼
  ├─▶ [验证阶段] ──▶ 验证结果
  │                     │
  │                     ▼
  ├─▶ [学习阶段] ──▶ 反思并记录
  │                     │
  │                     ▼
  └─▶ [LOOP_START] ◀── 永不停止，自动开始下一轮
```

### 循环条件

```
循环永远不会自然停止，除非:
1. 用户手动说"停止"或"停止循环"
2. 开发环境完全崩溃
3. 所有可完成的任务都已完成 (理论极限)

即使所有Sprint完成，循环仍然继续:
- 寻找代码优化机会
- 发现新的改进点
- 完善文档
- 优化测试覆盖
- 研究新技术

→ 真正的闭环: 循环永不终止
```

## 2. 感知阶段 (Sense) - 自动触发

### 每次循环自动执行

```bash
# 1. Git状态检查
git status --short

# 2. 编译状态检查
mvn compile -q 2>&1 | tail -20

# 3. 测试状态检查
mvn test -q 2>&1 | tail -20

# 4. 服务健康检查
curl -s http://localhost:8080/api/sprite/state 2>/dev/null || echo "服务未运行"

# 5. 日志检查
tail -30 ~/sprite.log 2>/dev/null | grep -E "ERROR|WARN" || echo "无异常日志"
```

### 感知结果判断

```
感知结果:
├─ 有未提交代码?     ──▶ [执行阶段] 提交代码
├─ 编译失败?        ──▶ [执行阶段] 修复编译错误
├─ 测试失败?        ──▶ [执行阶段] 修复测试
├─ 服务崩溃?        ──▶ [执行阶段] 重启服务
├─ 有待办任务?      ──▶ [规划阶段] 选择任务
└─ 一切正常?        ──▶ [寻找改进] 探索优化机会
```

## 3. 分析阶段 (Analyze)

### 自动分析维度

```
分析清单 (每次循环必做):
├─ 代码分析
│   ├─ 核心模块健康度
│   ├─ 技术债务
│   └─ 依赖安全
│
├─ 架构分析
│   ├─ 耦合度
│   ├─ 扩展性
│   └─ 性能瓶颈
│
├─ 流程分析
│   ├─ 开发效率
│   ├─ 重复工作
│   └─ 自动化机会
│
└─ 需求分析
    ├─ 遗漏点
    └─ 新需求
```

### 分析输出

```
每次分析输出:
├─ 问题列表 (Problem List)
├─ 改进机会 (Improvement Opportunities)
├─ 新增需求 (New Requirements)
└─ 风险评估 (Risk Assessment)
```

## 4. 规划阶段 (Plan)

### 任务优先级

```
使用 ICE 模型:
优先级分数 = Impact × Confidence × Ease

选择最高分任务执行
```

### 任务来源

```
任务发现渠道:
├─ 问题驱动 (Bug/性能/安全)
├─ 价值驱动 (新功能/增强)
├─ 改进驱动 (重构/优化)
└─ 探索驱动 (调研/原型)
```

### 任务执行

```
每个任务:
├─ 1-4小时完成
├─ 可独立验证
├─ 可单独提交
└─ 不阻塞其他任务
```

---

## 5. 执行阶段 (Execute)

### 执行规则

```
执行时:
├─ 先读代码再改
├─ 最小改动原则
├─ 边做边测
└─ 保持节奏，不卡死
```

### 遇到阻塞

```
阻塞处理:
├─ 环境问题  → 记录并跳过
├─ 依赖缺失  → 记录并跳过
├─ 权限问题  → 记录并跳过
└─ 继续下一个任务
```

---

## 6. 验证阶段 (Verify)

### 验证清单

```
验证项:
├─ 编译验证 (mvn compile)
├─ 测试验证 (mvn test)
├─ 功能验证 (新功能正常)
├─ 代码质量 (无坏味道)
└─ Git验证 (已提交)
```

### 验证结果

```
验证成功 → 更新文档 → 提交 → LOOP_START

验证失败 → 分析原因 → 修复 → 重新验证
                              ↑
                        (最多重试3次)
                              │
                              ▼
                        记录问题 → LOOP_START
```

## 11. 循环核心机制

### 自主驱动模式

```
当以下条件满足时，自动开始新循环:

1. [任务完成]    → 验证 → 提交 → LOOP_START
2. [验证失败]    → 修复 → 重试(≤3次) → LOOP_START
3. [一切正常]    → 寻找改进 → LOOP_START
4. [用户触发]    → 任意时刻用户可介入
```

### 永不停止的条件

```
循环停止条件:
✗ 不因"任务完成"停止
✗ 不因"没有任务"停止
✗ 不因"一切正常"停止
✗ 不因"文档齐全"停止

✓ 只因用户明确说"停止"而停止
✓ 只因环境崩溃而停止
✓ 只因所有可能的改进都穷尽而停止 (理论)
```

### 自动检测循环

```
循环检测器 (每次循环开始):
├─ 检查 git status
│   └─ 有改动 → 执行 → 提交 → LOOP
│
├─ 检查 mvn compile
│   └─ 编译失败 → 修复 → LOOP
│
├─ 检查需求池
│   └─ 有待办 → 执行最高优先级 → LOOP
│
└─ 检查改进机会
    └─ 有优化点 → 执行 → LOOP
        │
        └─ 无优化点 → 简单检查 → LOOP
```

## 12. 执行流程 (完整版)

```
启动:
  │
  ▼
[感知] ──── 检查项目状态
  │
  ▼
[分析] ──── 发现问题和机会
  │
  ▼
[规划] ──── 选择最高优先级任务
  │
  ▼
[执行] ──── 完成编码
  │
  ▼
[验证] ──── 编译 + 测试
  │       │
  │       ├─ 成功 → [文档同步]
  │       │
  │       └─ 失败 → [修复] → [重试验证] (最多3次)
  │                       │
  │                       └─ 仍失败 → [记录问题] → [文档同步]
  │
  ▼
[文档同步] ── 更新 PROGRESS / DEMAND_POOL / HANDOVER
  │
  ▼
[Git提交] ── 提交代码
  │
  ▼
[学习反思] ── 记录经验教训
  │
  ▼
[LOOP_START] ◀──────────────────────────────────────
  │
  └─────────────────────────────────────────────────
                      永不停止
```

## 13. 紧急情况处理

```
异常处理:
├─ 编译失败   → 分析错误 → 修复 → 验证 → 继续
├─ 测试失败   → 定位问题 → 修复 → 验证 → 继续
├─ 服务崩溃   → 检查原因 → 重启 → 验证 → 继续
└─ 性能问题   → Profiler → 优化 → 验证 → 继续
```

## 14. 心理健康

```
自我保护:
├─ 任务不堆积 (保持3-5个待办)
├─ 困难及时跳过 (不卡死)
├─ 定期反思 (避免重复错误)
└─ 保持节奏 (稳定持续 > 突击)
```
